# 多 Agent 适用边界

## 问题

什么时候应该引入多 agent，而不是继续优化单 agent？

## 当前结论

- 默认先做单 agent，只有在明确约束出现后再引入多 agent。
- 多 agent 的收益点主要来自上下文隔离、并行覆盖和任务专门化，不来自“结构看起来更高级”。
- 多 agent 的主要代价是上下文复制、协调开销、结果摘要和路由复杂度。

## 判断顺序

### 第一步：先怀疑是不是单 Agent 还没做好

- 提示词、工具设计、上下文压缩、记忆策略、权限边界没做好时，很容易把“系统设计问题”误判成“必须上多 agent”。
- 文章里最有价值的提醒不是多 agent 模式本身，而是默认起点应该保守。
- 如果单 agent 还没有做到稳定、可观测、可验证，直接扩成多 agent 往往只会把问题复制到更多节点。

### 第二步：确认瓶颈是否真的来自上下文或任务结构

- 如果主问题是上下文被无关信息污染，多 agent 才适合作为隔离手段。
- 如果任务天然可以拆成多个互不依赖的研究面，多 agent 才适合作为并行覆盖手段。
- 如果核心问题是工具集合过大、角色约束冲突、领域知识差异过大，多 agent 才适合作为专门化手段。

### 第三步：确认 handoff 是否值得

- 每多一个 agent，就多一次 handoff。
- 每次 handoff 都会带来摘要损失、路由错误、状态不一致和 token 成本。
- 如果拆分后仍需频繁同步上下文，通常不值得拆。

## 适合引入多 Agent 的三类场景

### 1. 上下文污染明显

- 某些子任务会产生大量主任务并不需要保留的上下文。
- 典型例子是检索、查单、日志分析、长文档过滤。
- 这类任务适合让子 agent 在独立上下文中处理，再只把压缩结果返回主 agent。
- 关键不是“把工作分出去”，而是“把无关上下文挡在主 agent 外面”。

### 2. 任务天然可并行

- 一个问题可以拆成多个相互独立的研究面、检索面或验证面。
- 并行的主要收益是扩大搜索覆盖度，不一定减少总 token，也不一定缩短总耗时。
- 对研究、搜索、横向比较类问题，这种方式通常提升的是覆盖率和稳健性，而不是响应速度。

### 3. 明显需要专门化

- 工具很多且跨域，单 agent 容易选错工具。
- 不同子任务需要冲突的系统提示或人格约束。
- 某些任务需要较重的领域知识，不适合全塞进一个通用 agent。
- 专门化不是为了“拟人化分工”，而是为了减少工具混淆、提示冲突和领域负担。

## 验证 Agent 为什么更稳

- verification subagent 之所以是文章中特别稳的一类模式，不是因为它“更工程化”，而是因为它需要的上下文最少。
- 它面对的是清晰的输入件、清晰的成功标准和相对黑盒的验证目标。
- 这意味着它规避了多 agent 最容易失败的信息传递问题。

## 对 coding agent 设计的启发

- 不要把多 agent 当作默认层级。
- 先把单 agent 的上下文治理、工具边界、状态管理做扎实。
- 只有当子任务边界足够清晰，且 handoff 真的能减少主链路负担时，再引入多 agent。
- 在代码场景里，最差的拆法通常是把同一功能的规划、实现、测试、评审拆成不同 agent。

## 不适合的拆分方式

- 按“规划 -> 实现 -> 测试 -> 评审”这类阶段机械拆分同一工作。
- 把紧耦合组件强行拆成多个 agent。
- 让多个 agent 反复同步同一份共享状态。

## 经验法则

- 拆分边界应按上下文边界，而不是按岗位边界。
- 如果 handoff 会持续损失信息，拆分就是负收益。
- verification subagent 是相对稳健的模式，因为它更多是黑盒验证，不依赖完整实现上下文。
